The Twelve-Factor App

The twelve-factor app is a methodology for building software-as-a-service apps that use declarative formats for setup automation, have a clean contract with the underlying operating system, are suitable for deployment on modern cloud platforms, and can scale up without significant changes to tooling, architecture, or development practices.

Introduction

In the modern era, software is commonly delivered as a service: called web apps, or software-as-a-service. The twelve-factor app is a methodology for building SaaS apps that:

  • Use declarative formats for setup automation
  • Have a clean contract with the underlying operating system
  • Are suitable for deployment on modern cloud platforms
  • Minimize divergence between development and production
  • Can scale up without significant changes to tooling, architecture, or development practices

I. Codebase

One codebase tracked in revision control, many deploys. A twelve-factor app is always tracked in a version control system, such as Git, Mercurial, or Subversion. A copy of the revision tracking database is known as a code repository, often shortened to code repo or just repo.

Mercurial
Subversion

Key Concepts:

Codebase

Any single app is always going to have a one-to-one correlation between the codebase and the app.

Deploy

A deploy is a running instance of the app. This is typically a production site, and one or more staging sites.

II. Dependencies

Explicitly declare and isolate dependencies. A twelve-factor app never relies on implicit existence of system-wide packages. It declares all dependencies, completely and exactly, via a dependency declaration manifest.

npm
pip

Key Concepts:

Dependency Declaration

A manifest that lists all dependencies required by the application, including exact version numbers.

Dependency Isolation

Using tools to ensure that no implicit dependencies leak in from the surrounding system.

III. Config

Store config in the environment. An app's config is everything that is likely to vary between deploys (staging, production, developer environments, etc). Apps sometimes store config as constants in the code, which is a violation of twelve-factor.

Key Concepts:

Environment Variables

Configuration values stored in the environment, easily changed between deploys without changing any code.

Config Separation

Strict separation of config from code ensures that config varies substantially across deploys, code does not.

IV. Backing Services

Treat backing services as attached resources. A backing service is any service the app consumes over the network as part of its normal operation. Examples include datastores, messaging/queueing systems, SMTP services for outbound email, and caching systems.

Key Concepts:

Backing Service

Any service the app consumes over the network as part of its normal operation.

Attached Resource

Services that are accessed via URL or other locator/credentials stored in config.

V. Build, Release, Run

Strictly separate build and run stages. A twelve-factor app uses strict separation between the build, release, and run stages. The build stage transforms a code repo into an executable bundle known as a build.

Key Concepts:

Build Stage

Transforms a code repo into an executable bundle by fetching vendors dependencies and compiling binaries and assets.

Release Stage

Takes the build and combines it with the deploy's current config.

Run Stage

Runs the app in the execution environment by launching some set of the app's processes against a selected release.

VI. Processes

Execute the app as one or more stateless processes. Twelve-factor processes are stateless and share-nothing. Any data that needs to persist must be stored in a stateful backing service, typically a database.

Key Concepts:

Stateless Process

Processes that store no persistent state and assume nothing about what's stored in memory or on disk.

Share-Nothing Architecture

Processes that never assume that anything cached in memory or on the filesystem will be available on a future request or job.

VII. Port Binding

Export services via port binding. The twelve-factor app is completely self-contained and does not rely on runtime injection of a webserver into the execution environment to create a web-facing service.

Key Concepts:

Port Binding

The web app exports HTTP as a service by binding to a port and listening to requests coming in on that port.

Self-Contained

The app does not rely on runtime injection of a webserver into the execution environment.

VIII. Concurrency

Scale out via the process model. In the twelve-factor app, processes are a first class citizen. Processes in the twelve-factor app take strong cues from the unix process model for running service daemons.

Key Concepts:

Process Model

Apps can span multiple processes running on multiple physical machines using the process model for scale.

Process Types

Different types of work assigned to different process types, such as web processes, worker processes, etc.

IX. Disposability

Maximize robustness with fast startup and graceful shutdown. The twelve-factor app's processes are disposable, meaning they can be started or stopped at a moment's notice.

Key Concepts:

Disposable Processes

Processes that can be started or stopped at a moment's notice for fast elastic scaling, rapid deployment, and robust production deploys.

Graceful Shutdown

Processes should handle termination gracefully when they receive a SIGTERM signal.

X. Dev/Prod Parity

Keep development, staging, and production as similar as possible. The twelve-factor app is designed for continuous deployment by keeping the gap between development and production small.

Key Concepts:

Environment Parity

Keeping development, staging, and production environments as similar as possible in terms of time, personnel, and tools.

Continuous Deployment

A software development practice where code changes are automatically deployed to production after passing tests.

XI. Logs

Treat logs as event streams. A twelve-factor app never concerns itself with routing or storage of its output stream. It should not attempt to write to or manage logfiles.

Key Concepts:

Event Stream

Logs are treated as a stream of aggregated, time-ordered events collected from the output streams of all running processes and backing services.

Log Aggregation

In staging or production deploys, each process' stream will be captured by the execution environment and collated together with all other streams.

XII. Admin Processes

Run admin/management tasks as one-off processes. The process formation is the array of processes that are used to do the app's regular business. Separately, developers will often wish to do one-off administrative or maintenance tasks for the app.

Key Concepts:

One-off Processes

Administrative processes that should be run in an identical environment as the regular long-running processes of the app.

Admin Tasks

Running database migrations, running a console, or running one-time scripts committed into the app's repo.

Implementation Benefits

The twelve-factor methodology provides numerous benefits for modern application development:

  • Portability: Applications can run on any cloud platform
  • Scalability: Applications can scale horizontally with minimal changes
  • Maintainability: Clear separation of concerns makes code easier to maintain
  • Reliability: Stateless processes and proper error handling improve reliability
  • Deployment: Continuous deployment becomes possible and safe

The Twelve-Factor App methodology was created by Adam Wiggins, co-founder of Heroku. It draws on experience with hundreds of Software-as-a-Service applications and represents a synthesis of best practices for app development.
